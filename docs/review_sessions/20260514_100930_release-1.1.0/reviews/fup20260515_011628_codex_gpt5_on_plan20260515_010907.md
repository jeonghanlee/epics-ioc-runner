## Findings

### F-PLAN4-1. sudoers boundary is overstated as an `ioc-runner` execution gate

Severity: blocking for Step 0-R + Step 4-R acceptance.

The revised permission model is technically coherent if the sudoers layer is described as the gate for privileged system-mode state changes. However, the current plan, test plan, handoff, and `LOG_PERMISSIONS.md` repeatedly describe sudoers as if it restricts `ioc-runner` execution or all system-mode operations to `%ioc`. That is broader than the code actually enforces.

Evidence:

- `bin/setup-system-infra.bash` emits sudoers rules only for `/usr/bin/systemctl start|stop|restart|status|enable|disable|daemon-reload` on `epics-@*.service`.
- `bin/ioc-runner` does not check group membership before executing.
- `bin/ioc-runner` calls `systemctl status`, `cat`, `show`, and `is-active` without `sudo` in system mode.
- Therefore sudoers does not prevent a non-`ioc` user from invoking `ioc-runner` itself, nor does it necessarily block read-only status-style paths that do not require sudo on the host.

Affected text:

- `plan20260515_010907` says "Non-`ioc` users cannot run `ioc-runner` (sudoers layer)".
- `docs/TEST_PLAN-1.1.0.md` says "`ioc-runner` execution restricted to `ioc` group by sudoers".
- `docs/LOG_PERMISSIONS.md` says "Only members of the `ioc` group can run `ioc-runner` system-mode operations" and "the sudoers policy already prevents them from using `ioc-runner` itself".
- `hand20260515_010907` says the sudoers gate restricts `ioc-runner` system-mode operations to `%ioc`.

Required correction:

Revise the model wording to say that sudoers gates privileged state-changing `systemctl` operations issued by `ioc-runner` in system mode. Do not claim it gates `ioc-runner` process execution itself unless a separate explicit group-membership guard is added to `ioc-runner`.

Verification impact:

The negative probe should target a privileged operation, for example non-`ioc` `sudo /usr/bin/systemctl start epics-@<name>.service`, not generic `ioc-runner` execution or read-only `status`. A policy-text/`visudo` check is acceptable only if the documented assertion is narrowed to the sudoers rule's actual command surface.
