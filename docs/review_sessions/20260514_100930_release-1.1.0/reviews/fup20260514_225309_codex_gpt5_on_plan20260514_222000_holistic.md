### F-PLAN-2 — P-B-2 does not state how existing local templates are updated

State: revision required before UD006

Maps to: R-PLAN-4, R-PLAN-8

Evidence:

- `bin/ioc-runner` `deploy_local_template` only emits
  `${SYSTEMD_DIR}/epics-@.service` when the file does not already exist
  (`bin/ioc-runner:312-335`).
- `do_install` calls `deploy_local_template` in local mode
  (`bin/ioc-runner:872-882`), but does not force regeneration of an
  existing user template.
- `plan20260514_222000` requires the P-B-2 local user unit to gain
  `UMask=0027` and later V-B-2 to assert that directive, but it does
  not specify whether the implementation updates existing local
  templates, refuses with a migration hint, or verifies only a fresh
  no-template install.

Impact:

A developer can implement the heredoc change correctly and still have
V-B-2 fail or become environment-dependent on a host that already has
`~/.config/systemd/user/epics-@.service` from an older release.

Required revision:

- Add an explicit P-B-2 template-update policy for existing local user
  templates. Acceptable forms include a safe overwrite path, a content
  check plus regeneration path, or a documented verification precondition
  that deletes the existing generated template before testing.
- Update V-B-2 to include that precondition or update behavior, plus
  `systemctl --user daemon-reload` after the template changes.

### F-PLAN-3 — Recovery Boundary mixes pre-commit failure with post-commit revert

State: revision required before UD006

Maps to: R-PLAN-7

Evidence:

- `plan20260514_222000` lines 263-270 say that if
  P-B-1 + P-B-2 + P-C2 execution fails verification on `top`, recovery
  is by `git revert` of the single combined commit.
- Under the session gate model, implementation and verification happen
  before User phase commit. A verification failure during Implementer
  work normally has no committed milestone to revert.
- The same work also mutates runtime deployment state outside git:
  `/etc/systemd/system/epics-@.service`, user
  `~/.config/systemd/user/epics-@.service`, daemon-reload state, and
  `/var/log/procserv` runtime files.

Impact:

The recovery text is directionally useful for a post-commit rollback,
but it is not realistic as the primary recovery path for pre-commit
verification failure on `top`.

Required revision:

- Split recovery into pre-commit and post-commit cases.
- Pre-commit failure: repair or abandon the working-tree changes, then
  restore generated system/user unit files as needed and reload the
  appropriate systemd manager.
- Post-commit rollback: User may `git revert` the combined commit, then
  rerun the reverted generator or restore prior generated units; note
  that `/var/log/procserv` and existing log files may persist and must
  be inspected rather than assumed removed.
- Keep the manual local-template deletion note, but make it part of the
  broader generated-unit rollback procedure.
