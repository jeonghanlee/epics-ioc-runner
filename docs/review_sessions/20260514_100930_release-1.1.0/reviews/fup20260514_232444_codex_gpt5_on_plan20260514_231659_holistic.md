## Findings

### F-PLAN2-1 — Setgid alone does not satisfy the stated creation-order `rw` invariant

State: revision required before UD006

Maps to: R-PLAN2-2, R-PLAN2-5, R-PLAN2-8

Evidence:

- `plan20260514_231659` states that the three-principal invariant is
  "all three principals ... can read AND write any log file regardless
  of creator" (lines 85-87).
- The same plan states that setgid on `${SYSTEM_LOG_DIR}` forces group
  `ioc`, and with mode `0660` both `ioc-srv` and engineers retain `rw`
  "whichever one created the file first" (lines 134-138).
- The planned mechanism only controls file mode for procServ-created
  files through systemd `UMask=0007` (lines 115 and 202). It does not
  control file mode for engineer-created files, copied files, or files
  restored with preserved modes.
- A local setgid-directory probe with process umask `0022` created a
  file mode `0644`, demonstrating that setgid controls group
  inheritance but not group write mode.
- Existing site docs already distinguish the shared IOC tree as
  `root:ioc`, mode `2775`, and default ACLs:
  `docs/FAQ.md:161` and `docs/USER_GUIDE.md:22`.

Impact:

If an engineer creates or copies `${SYSTEM_LOG_DIR}/<ioc>.log` before
procServ does, the group may be `ioc`, but group write can still be
absent. Then `ioc-srv` cannot append to the file despite the plan's
creation-order invariant.

Required revision:

- Choose one of the following and record it explicitly:
  - Narrow the invariant to "procServ is the only supported creator for
    `<ioc>.log`; engineers may read/write files after procServ creates
    them", then remove the "regardless of creator" claim.
  - Add default ACLs to the log directory plan, for example default
    group and mask entries that preserve group `rw` for newly created
    files, and add `setfacl` / `getfacl` availability and verification
    steps for Debian 13 and Rocky 8.
- Extend V-C2 to test the selected behavior. If creation-order
  invariance remains a requirement, V-C2 needs an engineer-created
  preexisting log file case, followed by procServ append verification.

### F-PLAN2-2 — Local user-template backup name can collide

State: revision required before UD006

Maps to: R-PLAN2-4

Evidence:

- `plan20260514_231659` requires always overwriting the local user unit
  and backing up an existing template to
  `${template_path}.bak-<timestamp>` with timestamp format
  `YYYYMMDD-HHMMSS` (lines 254-260).
- A seconds-resolution timestamp is not unique for repeated installs in
  the same second, nor for concurrent installs against a shared home
  directory.
- The system-side `backup_if_exists` function also uses a timestamp,
  but it writes under a root-owned backup directory for a root-run
  setup script; the proposed local backup path is per-user and may live
  on shared or NFS-backed home storage.

Impact:

The backup step can overwrite or race with another backup, which defeats
the plan's stated preservation guarantee for customized or prior local
templates.

Required revision:

- Use a collision-resistant backup path, such as `mktemp` in the same
  target directory, or include at least timestamp + PID and fail with
  `noclobber` semantics if the target already exists.
- Add V-B-2 evidence that a prior template is preserved and that
  repeated installs produce distinct backups.
