## Confirmed Findings

### Q-PERM-1 — P-C2 directory ownership target is incorrect

State: accepted

Reviewer 1 concurs that the P-C2 plan target
`/var/log/procserv/ = root:ioc-srv 0750` is inconsistent with the
repository's current system-mode account model.

Evidence:

- `bin/setup-system-infra.bash:16-17` defines
  `SYSTEM_USER="ioc-srv"` and `SYSTEM_GROUP="ioc"`.
- `bin/setup-system-infra.bash:360-361` emits `User=${SYSTEM_USER}`
  and `Group=${SYSTEM_GROUP}` into the system unit.
- Local `systemd.exec(5)` documents that `LogsDirectory=` directories
  are created under `/var/log/`, and except for
  `ConfigurationDirectory=`, the innermost directories are owned by
  the unit `User=` and `Group=`.

Required revision:

- Correct the system directory target to `ioc-srv:ioc 0750`.
- Remove the planned `chgrp ioc-srv /var/log/procserv` step. That
  group change would remove directory traversal from engineers in the
  `ioc` group and conflict with the Manage principal requirement.
- Keep P-C2 focused on verifying the post-activation ownership and
  mode rather than overriding the unit-created directory.

### Q-PERM-2 — Log file mode 0640 needs UMask=0027 in P-B-1

State: accepted

Reviewer 1 concurs that `UMask=0027` belongs in the system unit emitted
by P-B-1. The log file mode is created by the procServ process and
there is no procServ option in `procServ --help` for log file mode or
process umask.

Evidence:

- `procServ --help` exposes `-L --logfile <file>` but no logfile mode
  or umask option.
- Local `systemd.exec(5)` documents `UMask=` as the file mode creation
  mask and states that system units default to `0022`.
- A process creating a new log file under the default system unit
  umask can produce `0644`, which does not satisfy the planned `0640`
  target.

Required revision:

- Add `UMask=0027` to the P-B-1 system unit heredoc.
- Extend V-B-1 to assert `UMask=0027`.
- Keep V-C2 responsible for observing the resulting log file mode.

## Revised Finding

### Q-PERM-3 — Do not add LogsDirectoryGroup=

State: revision required

Reviewer 1 does not concur with adding `LogsDirectoryGroup=${SYSTEM_GROUP}`
to the emitted P-B-1 unit. The local Debian 13 systemd 257
`systemd.exec(5)` page contains `LogsDirectory=`,
`LogsDirectoryMode=`, `User=`, and `Group=` ownership semantics, but no
`LogsDirectoryGroup=` directive. A direct search of the same local
manual also found no `RuntimeDirectoryGroup=`, `StateDirectoryGroup=`,
`CacheDirectoryGroup=`, or `LogsDirectoryGroup=` family.

The desired dependency should be made explicit in the plan and
verification text, not by adding an unsupported unit directive.

Required revision:

- Do not add `LogsDirectoryGroup=` to the system unit.
- Keep `Group=${SYSTEM_GROUP}` in the unit as the ownership source for
  `LogsDirectory=procserv`.
- Update the plan text to state that `LogsDirectory=procserv` relies on
  `User=${SYSTEM_USER}`, `Group=${SYSTEM_GROUP}`, and
  `LogsDirectoryMode=0750`, with current values yielding
  `ioc-srv:ioc 0750`.
- Extend V-B-1 to assert the unit contains `Group=ioc`,
  `LogsDirectory=procserv`, `LogsDirectoryMode=0750`, `UMask=0027`,
  and `--logfile=/var/log/procserv/%i.log`.
- Extend V-C2 to assert the realized filesystem state:
  `/var/log/procserv` is `ioc-srv:ioc 0750`, and
  `/var/log/procserv/<ioc>.log` is `ioc-srv:ioc 0640`.
